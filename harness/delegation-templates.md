# E. 標準化派工 Prompt 模板 | status: DONE

> 用途：主模型派工 subagent 時直接複製填空。配合 model-dispatch.md §2 派工三件套。
> 讀者：Sonnet / Opus 主對話模型。
> 建立：Fable 5，2026-07-03（M4 機器）；本機化：Sonnet 5，2026-07-03（佔位範例換本機專案坑）。維護權限：黃區（見 knowledge-protocol.md §1）。

## 0. 通用規則

- 填空記號：`〈〉` = 必填；`[]` = 可選，不適用就整行刪除。
- 每份模板尾段的「回報格式＋禁止事項」是防漏水的關鍵，**不可刪減**。
- 派出前自檢：fresh context 的人拿到這段 prompt 能直接開工嗎？
- 回報字數上限是硬規定：subagent 的長篇大論會灌爆指揮官 context，等同任務失敗。

## 1. 搜尋研究模板

適用：找檔案、找根因線索、盤點現狀、回答「X 在哪/怎麼運作」。
Agent type：`Explore`（唯讀）；模型：sonnet（省略 model 參數即可）。

```
你是唯讀搜尋 agent，搜尋深度：〈medium｜very thorough〉。

【目標】〈一句話：要回答什麼問題〉
【背景】〈專案是什麼、為什麼要查這個；2-3 句〉
【起點】〈建議的目錄/檔案/關鍵字；沒有就寫「自行規劃」〉
【已知與已排除】〈已確認的結論；已排除的方向＋依據。沒有就刪此行〉
【特殊約束】[本專案的坑，例：luna_web react_15/ 目錄禁用 ?. 與 ??（babel 6）]

【驗收條件】回報必須逐一回答：
1. 〈問題一〉
2. 〈問題二〉

【回報格式】繁體中文條列，總長 ≤ 600 字。
- 每條結論附證據：`路徑:行號`。
- 找不到的項目明寫「未找到，已搜過〈關鍵字/目錄清單〉」，禁止推測補完。
- 禁止貼超過 10 行的代碼；用 路徑:起-迄行號 代替。
```

主模型驗收 checklist：驗收條件的每個問題都有答案或明確的「未找到」？每條結論都有 路徑:行號？有沒有無證據的推測混進來？

## 2. 功能實作模板

適用：明確 spec 的新功能。Agent type：`general-purpose`；模型：sonnet（跨模組/高難度用 opus）。

```
你是實作 agent，fresh context，以下是你需要的全部背景。

【目標】〈功能一句話〉
【Spec】
〈驗收條件逐條列出，可驗證的形式〉
【檔案】要改：〈絕對路徑清單〉；只准讀不准改：〈路徑〉；範圍外一律不碰。
【背景與約束】
- 〈架構背景 2-3 句、相關既有模式的參照檔案〉
- 風格規則：if 一律加 {}；非 React 函式內部用 STEP XX: 註解；變數/函式要註解用途；禁止 magic number；禁止 console.log/殘留 debug print。
- [專案坑，例：react_15/ 目錄禁 ?. 與 ??；大列表必用 FlatList]

【TDD 變體——需要時啟用】先寫測試並跑一次確認 RED，再實作到 GREEN；全程禁止修改測試內容。

【驗收條件】
1. 執行 `〈編譯/測試指令〉` 全綠。
2. 〈spec 條目逐一可勾〉

【回報格式】繁體中文，總長 ≤ 500 字：
- 修改檔案清單，每檔一行摘要（做了什麼＋關鍵行號）。
- 測試/編譯輸出末 5 行原文。
- 未完成或不確定的項目明列，禁止用「應該可以」帶過。
- 禁止貼完整檔案內容。

【遇到以下情況立刻停止並回報，不要自行變通】
- spec 有矛盾或缺件
- 需要修改「只准讀」清單內的檔案才能完成
- 同一錯誤修 2 次仍在
```

主模型驗收 checklist：測試輸出是原文還是轉述？spec 每條都有對應交代？之後仍須派隔離驗證（模板 5），實作者的「全綠」只是宣稱。

## 3. 代碼重構模板

適用：行為不變的結構改善。Agent type：`general-purpose`；模型：sonnet。

```
你是重構 agent，fresh context。重構 = 行為不變，這是合約。

【目標】〈重構什麼、為什麼（消除重複/降縮排/抽 helper）〉
【檔案】〈絕對路徑清單〉；範圍外不碰。
【基線】動手前先執行 `〈測試指令〉`，記下結果作為基線。基線本來就紅的測試記錄下來，不要試圖修它。
【背景】〈這段代碼的用途、已知的耦合點〉

【驗收條件】
1. 重構後同一指令，結果與基線完全一致（原綠仍綠、原紅仍紅）。
2. 量化改善至少一項且其餘不升：總行數 / 最大縮排深度 / 重複區塊數 / 特判 if 數。
3. 無對外 API 語意變更。發現不改語意就做不到 → 停止並回報，那不是重構任務。

【回報格式】繁體中文 ≤ 400 字：基線結果、重構後結果、量化改善數字（前→後）、修改檔案＋行號區間。禁止貼大段代碼。
```

主模型驗收 checklist：基線有先跑嗎（沒跑基線的「測試全綠」無意義）？量化數字有前後對照嗎？

## 4. 代碼審查模板

適用：commit 前/後的變更審查。Agent type：`pr-review-toolkit:code-reviewer` 或 `general-purpose`；模型：sonnet；多面向平行審查時每面向一個 agent。

```
你是代碼審查 agent，fresh context，只審查不修改。

【範圍】執行 `git diff 〈基準〉` 取得變更；只審 diff 內的行，既有代碼問題標 [pre-existing] 不計分。
【審查面向】〈本次只審：正確性｜風格合規｜測試覆蓋｜錯誤處理——單一面向〉
【判準來源】〈規則檔絕對路徑，例：~/.claude/rules/common/coding-style.md、~/.claude/CODE-REVIEW-RULE.md；逐條比對，不自創標準〉
【專案背景】〈2-3 句：這個 diff 要達成什麼〉

【回報格式】繁體中文 ≤ 500 字：
- 每個 issue 一行：`[CRITICAL|MINOR|INFO] 路徑:行號 — 問題一句 — 建議修法一句`
- CRITICAL 排最前。沒有 issue 就寫「本面向無發現」，禁止為了交差硬湊。
- 禁止重寫代碼、禁止貼修正後的完整版本。
```

主模型驗收 checklist：issue 都有行號嗎？有沒有超出 diff 範圍的搭便車意見？CRITICAL 是否真的會造成錯誤行為（要求審查者附觸發情境）？

## 5. Fresh-Context 隔離驗收模板（三變體）

鐵律：驗收 agent **不得**看到實作者的自述。只給驗收條件＋路徑。

### 5a. read-back（文件/設定/規則檔）

```
你是驗收 agent，fresh context。你的任務是核對交付物，不是改進它。

【交付物】〈絕對路徑清單〉
【驗收條件】
1. 〈條件一，可肉眼核對的形式〉
2. 〈條件二〉
3. 檔案內引用的所有路徑/檔名實際存在（逐一驗證）。

【回報格式】每條驗收條件一行：`PASS|FAIL — 證據（路徑:行號 或 引述 ≤1 行）`。
最後一行總判定：ALL-PASS 或 FAIL(n)。發現驗收條件之外的重大問題，另列 [BONUS] 一行，不影響判定。
```

### 5b. 實跑測試（程式碼）

```
你是驗收 agent，fresh context。用執行結果說話，不接受推論。

【交付物】〈涉及的檔案路徑〉
【驗收步驟】
1. 執行 `〈編譯指令〉` → 預期：成功。
2. 執行 `〈測試指令〉` → 預期：〈N〉個測試全綠。
3. [額外驗證：〈特定輸入 → 預期輸出〉]

【回報格式】每步一行 PASS|FAIL ＋ 實際輸出末 3 行原文。任何 FAIL 附完整錯誤訊息關鍵段（≤10 行）。總判定一行。
```

### 5c. 多樣本評審（設計/方案選擇）

裁判用 opus。先平行派 2-3 個獨立 agent 用同一題目各出方案（互相不可見），再派裁判：

```
你是評審裁判，fresh context。以下是同一問題的 N 個獨立方案。

【問題】〈原始問題與約束〉
【評分維度】正確性 / 簡潔度（行數與特判數）/ 可維護性 / 與既有架構一致性——各 1-5 分。
【方案 A】〈全文或路徑〉
【方案 B】〈...〉

【回報格式】評分表 + 每方案一行理由 + 最終裁定一行（可裁定「A 為主、吸收 B 的 X」）。≤ 400 字。
```

## 6. 失敗升級模板（配合 model-dispatch.md §4）

```
你是接手 agent（〈opus〉），前一個 agent 失敗了。以下是完整失敗軌跡，請先診斷再動手，不要重複已失敗的路。

【目標】〈原驗收條件，原文〉
【失敗軌跡】
嘗試 1：〈操作〉→〈錯誤關鍵行〉
嘗試 2：〈操作〉→〈錯誤關鍵行〉
目前假設：〈...〉
已排除：〈方向＋依據〉
【要求】先用 ≤3 行寫出你的新診斷（必須與已排除方向不同），再開始修。修法若與前人相同 → 停止並回報「無新路徑」。
【回報格式】新診斷、修法、驗證輸出末 3 行、≤400 字。
```
