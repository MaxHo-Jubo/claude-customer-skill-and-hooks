# D. 判斷力外化矩陣 | status: DONE

> 用途：把高階模型的直覺量化為可肉眼比對的檢核表。每條附【完美正例】【典型反例】。
> 讀者：Sonnet / Opus。三個使用時機：**卡關時（§1）、宣告完成前（§2）、想問 user 前（§3）**。
> 建立：Fable 5，2026-07-03（M4 機器）；本機化：Sonnet 5，2026-07-03（案例換為本機實例，硬體條款移除）。維護權限：黃區（見 knowledge-protocol.md §1）。

## 1. 換路徑信號（STOP-AND-PIVOT）

以下任一信號成立 = 方向可能錯了。標準動作永遠是：**停止修改 → 用一句話重寫「我認為的根因」→ 換一層診斷**（層次：程式邏輯 → 資料結構 → 環境/工具鏈 → 執行時實況），而不是在原地換個寫法再試一次。

### S1. 同一錯誤第三次出現
- 判準：同一錯誤訊息（或同一測試失敗）在 ≥ 3 次修改後仍出現，且其中至少兩次用了**不同**修法。
- 【完美正例】claude-mem 中文化三次修復（本機真實案例，2026-03-27）：前兩次 patch `cache/` 路徑，grep 確認檔案有中文字串但輸出仍是英文 → 停下，換層診斷——用 `ps` 查 worker daemon PID 的實際載入路徑 → 根因是 daemon 從 `marketplaces/` 載入程式碼，patch 的和跑的不是同一份檔案。檔案層修兩次無效，問題在執行時實況層。
- 【典型反例】測試一直紅，第 4、5、6 次繼續微調同一函式的邊界值，每次改一點點期待這次會過。

### S2. 蹺蹺板（修 A 壞 B）
- 判準：修好 A 導致 B 壞、修好 B 又導致 A 壞，來回 ≥ 2 個循環。
- 機制：A 與 B 共享的抽象/資料結構錯了，局部修補只是在搬問題。
- 【完美正例】兩個頁面共用同一個 reducer 欄位，修 A 頁顯示邏輯壞了 B 頁、反向修又壞回來 → 停止逐頁補丁，正解是把共享欄位拆開或建立「新增 state 必同步更新所有消費端」的結構性 checklist。
- 【典型反例】來回改兩個函式共 5 次，每次只讓其中一邊的測試變綠。

### S3. 特判累積
- 判準：一個修改需要新增 ≥ 3 個 if / 特例才能蓋住所有案例。
- 機制：if 是抽象未到位的症狀（taste 原則）。該改的是資料結構或視角，不是再加一個 if。
- 【完美正例】同一個「位置有效性」判斷 inline 散落在多個檔案，一處檢查空值、另一處沒檢查，行為不一致造成 fail-open bug（本機 extract-shared-helper 規則的由來）→ 正解是第一次出現重複時就抽具名共用 helper（`isValidLocation`）放 utility 檔，所有呼叫點統一引用。
- 【典型反例】第 4 個 `if (pageType == X && status == MAX)` 特判又出現，還繼續加第 5 個。

### S4. 解法要求違規
- 判準：唯一想得到的解法需要：跳過/刪除測試、修改測試遷就實作、hardcode 秘密、違反 user 已拍板的決策。
- 【完美正例】測試失敗 → 核對 spec 發現測試對、實作錯 → 修實作。
- 【典型反例】把 assert 的期望值改成實作的實際輸出，讓測試變綠。這不是修 bug，是銷毀證據。

### S5. 前提找不到
- 判準：按假設搜尋 3 輪（換過關鍵字、換過目錄）仍找不到預期存在的符號/檔案/設定。
- 機制：你對架構的想像是錯的。繼續搜是在搜一個不存在的東西。
- 【完美正例】用 codebase-memory-mcp `trace_path` 查某方法的 callers 回空 → 不能直接斷言「無人呼叫、可安全刪除」——`trace_path` 對「方法當 callback 參照傳遞」（React method 綁定後當 prop 傳出、Redux dispatch）是盲區 → 停止依賴單一工具，改用 grep 交叉驗證後才下結論（本機 graph-first 規則明文要求）。
- 【典型反例】第 5 輪換同義字繼續 grep。

### S6. diff 失控
- 判準：任務是 bugfix，但 diff 已超過 ~100 行或碰到 > 3 個檔案。
- 機制：不是 scope creep 就是根因誤判（真根因通常讓 diff 變小）。本機 GATE-1 bugfix-scope 同義：最小化改動，不順便重構、不擴大範圍到無法驗證的頁面。
- 【完美正例】發現修一個 bug 要動 6 個檔 → 停下重新診斷 → 真根因在共用 helper，一處修好，diff 12 行。
- 【典型反例】「順便」重構了三個無關函式，最後無法區分哪一行是修復本體。

## 2. 完成判準（DEFINITION-OF-DONE）

**通用前置**：宣告完成前，必須能寫出這句話——「我改了 X，因為根因/需求是 Y，證據是 Z（可指認的測試輸出/行號/實測結果）」。寫不出 Y 或 Z 就不是完成，是「感覺完成」。

### bugfix
- [ ] 根因可用一句話寫出（含機制，不是「改了就好了」）
- [ ] 有回歸測試；無法自動化驗證的（UI 行為、外部服務）至少有實測紀錄或截圖，做不到就明標「⏳ 待驗證＋驗證步驟」
- [ ] 全部既有測試綠（附輸出末 3 行）
- [ ] diff 只含修復本身，無順手重構、無無關格式改動
- [ ] 修復不得用 silent fallback 掩蓋（`?? defaultValue` 比 crash 更糟；治本後 shape 已收斂的值不加防護性 fallback——本機 no-fallback-after-root-cause 規則）

【完美正例】根因 = 共享狀態漏欄位導致更新被跳過；修法 = 補欄位；證據 = 新增回歸測試 + 全測試綠。
【典型反例】「我加了一個 100ms 的 setTimeout 之後就沒再重現」——無根因、無測試。這是 workaround，必須標註為 workaround 並留待查根因，不得標記完成。

### 新功能
- [ ] 驗收條件逐條對勾（引用原始需求原文，一條一勾）
- [ ] 新邏輯有測試；endpoint 改動先寫 integration test（supertest+jest，錯誤路徑 400/401/403/404/重複操作——本機 BACKEND-TEST 規則）
- [ ] 受影響文件同步（CLAUDE.md / README / spec 對應段落）
- [ ] 交付路徑內無 TODO / FIXME 殘留
- [ ] 全測試綠；E2E 截圖報告需符合「斷言截圖三合一規範」（rules/typescript/testing.md：程式邏輯斷言 + 真實頁面操作 + evidence overlay）

### 重構
- [ ] 行為不變證據：改動前後**同一組**測試皆綠（先跑基線再動手）
- [ ] 複雜度淨下降且可量化：行數 / 最大縮排深度 / 特判數至少一項下降、其餘不升
- [ ] 無對外 API 語意變更（有就不是重構，重新分類為功能變更並補走需求確認）

### 文件
- [ ] 文內引用的路徑/檔名逐一驗證存在
- [ ] 數字、行號、指令對照**當下**實際狀態（跑過/查過，不是憑記憶）
- [ ] 正體中文、無錯字、無過時段落殘留

## 3. 熔斷條件（CIRCUIT-BREAKER）

### 必熔斷（任一成立 → 停止自主作業，用下方格式問 user）
1. 動作不可逆且超出本次任務授權：刪檔/刪分支、force push、覆寫未 commit 的變更、rotate 憑證
2. 要動紅區檔案（settings.json、hooks、plugin 配置——見 knowledge-protocol.md §1）
3. 需求存在兩種合理解讀且成本差 > 2 倍（例：「列表改無限捲動」= 只改這一頁，還是全部使用同元件的頁面？）
4. 發現正確做法與 user 已明示的決策衝突
5. model-dispatch.md §4 升級狀態機兩輪走完仍失敗
6. 下一步需要 user 外部動作（提供測試帳號、正式環境操作確認、UI 截圖回饋）
7. 對外發布、花錢、寄信、開 PR 到別人的 repo、寫入 Jira/Confluence 正式內容

### 不該熔斷的反例（這些自己解決，問了 = 浪費 user 時間）
- 「這個 API 怎麼用」→ 查官方文件 / context7
- 「檔案在哪」→ 搜尋
- 「測試怎麼跑」→ 讀 package.json / README
- 「user 會不會想要 X」且 X 可逆、成本低 → 做可逆的預設選擇，在回報中標明「已預設選 X，可改」

### 熔斷提問標準格式
```
【卡點】一句話
【已試】嘗試與結果（≤ 3 行）
【選項】A：...（建議，理由一行）／ B：...
【影響】選錯的代價一行
```

## 4. 品味極限判準（TASTE-LIMIT）

### 識別信號（任一成立 = 品味決策，弱模型不得自行拍板）
- 驗收條件無法量化（「要有專業感」「要好看」「語氣要對」）
- 兩個候選都「正確」，差異只在審美 / 語感 / 品牌感
- UI 佈局、配色、文案風格的**從零**設計（非對照既有參照物）

### 標準應對（三選一，選了哪個要寫進回報）
1. **參照優先**：找已被 user 驗證過的參照物照抄結構。本機 UI 一律對齊 user 提供的截圖標註與既有頁面樣式（FRONTEND-ITERATION 節奏：第 1 輪實作 70-80% → 第 2 輪 user 截圖標註精調 → 第 3 輪微調），不另起爐灶。
2. **候選展示**：產出 2-3 個**差異明顯**的候選＋各自 tradeoff 一行，讓 user 選。禁止只出一個然後自評「我覺得不錯」。
3. **標註跳過**：在交付物中列出「品味決策點」清單留給 user，先完成可量化的部分。

【完美正例】要做新的進度報告 HTML → 直接沿用既有報告版型，只換內容。
【典型反例】自行發明新配色新字型，回報裡宣稱「更具現代感」——幻覺式審美，禁止。
