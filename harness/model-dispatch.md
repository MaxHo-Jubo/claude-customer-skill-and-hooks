# C. 模型調度與動態升降級守則 | status: DONE

> 讀者：主對話模型（Sonnet / Opus 等級），每次 spawn subagent 前照本檔執行。
> 所有規則均為可機械執行的判準。與 user 明示指令衝突時，user 指令優先。
> 建立：Fable 5，2026-07-03（M4 機器）；本機化：Sonnet 5，2026-07-03（新增 §6 本機工具鏈守則）。維護權限：黃區（見 knowledge-protocol.md §1）。

## 1. 指揮官不下場原則

主對話模型 = 指揮官。**指揮官的 context window 是全場最稀缺的資源**，把工具原始輸出灌進主對話 = 漏水。

指揮官只做五件事：
1. 需求確認與拆解（GATE-1）
2. 架構決策與方案選擇
3. 派工與驗收
4. 對 user 溝通
5. 最終整合（只合併 subagent 的**結論**，不合併原始輸出）

### 必須派工（機械判準，任一成立即派工）

| 判準 | 門檻 |
|------|------|
| 預期要讀的檔案數 | > 3 個 |
| 預期要讀的總行數 | > 400 行 |
| grep/搜尋且命中數未知 | 一律派 Explore |
| repo 全域掃描 / 盤點 / 審計 | 一律派工 |
| 外部文件 / 網路研究 | 一律派工 |
| log 分析（> 100 行） | 一律派工 |
| 多檔實作（要改 > 2 個檔） | 派 general-purpose |

### 指揮官可親自動手的例外（**全部**成立才可）

- 要讀的檔案路徑已知且 ≤ 3 個；要**改**的檔案 ≤ 2 個（與上表派工門檻對齊）
- 預期改動 ≤ 100 行（與 judgment-matrix.md §1 S6 的 diff 失控門檻對齊）
- 不需要探索式搜尋
- 典型：單檔 bugfix、typo、改設定值、跑一條測試指令、git 操作

【完美正例】user：「把 debounce 門檻從 50ms 改成 80ms」→ 路徑已知、單檔、直接改，不派工。
【典型反例】user：「找出按鈕為什麼有時雙觸發」→ 指揮官自己連續 Read 十個檔案 + 貼 log，context 燒掉一半才開始修。正解：派 Explore 收斂到 2-3 個嫌疑點，指揮官只收結論。

## 2. 派工三件套（缺一不派）

每次 Agent 呼叫的 prompt 必含三件套。模板見 [delegation-templates.md](delegation-templates.md)，直接複製填空。

1. **明確目標與背景**：subagent 是 fresh context，你知道的它全都不知道。必附：任務一句話、相關檔案絕對路徑、已知結論與已排除方向、專案特殊約束（例：luna_web `react_15/` 目錄禁用 `?.` 與 `??`，babel 6 不支援）。
2. **嚴格驗收條件**：可驗證的完成定義（「測試 X 綠」「回報必須回答問題 Y」），禁止「做好它」這種空話。
3. **標準回報格式**：明定結構與長度上限；一律要求「成果路徑 + 關鍵行號」；引用 > 20 行的代碼一律改寫成 `路徑:起-迄行號`，禁止噴大段代碼。

派工前自檢一句話：「一個從沒看過這個 repo 的人，拿到這段 prompt 能直接開工嗎？」不能就補件，不要先派再補。

## 3. 模型分工表

Agent 工具的 `model` 參數可填：`haiku` / `sonnet` / `opus`（本機另有 `fable`）。省略 = 繼承主對話模型（多數情況正確）。

| 任務型態 | 模型 | 理由 |
|---------|------|------|
| 已解出固定模式的批次套用（如 20 個檔套同一 pattern） | haiku | 模式已定，純機械 |
| 格式整理、單檔摘要、樣板生成 | haiku | |
| 搜尋研究、明確 spec 的實作、有測試保護的重構、checklist 式審查 | sonnet | 預設工作馬 |
| 架構設計、模糊除錯、跨模組影響分析、多樣本評審的裁判 | opus | |
| 升級狀態機的終點（§4） | opus | |

**Haiku 禁區**（即使看起來簡單也不派 Haiku）：需要多步工具串接的除錯、需要理解跨檔案 invariant 的修改、任何「找原因」類任務。

## 4. 異常升降級路徑（狀態機）

「一次錯誤」的機械定義：工具呼叫格式錯 / 編譯失敗 / 測試失敗 / 回報不符驗收條件 / 明顯幻覺（引用不存在的路徑或符號）。

```
Haiku 錯 1 次 ──────────────→ 同任務改派 Sonnet（不給 Haiku 第二次機會）
Sonnet 同一子任務連錯 2 次 ──→ 帶「失敗軌跡」改派 Opus
Opus 解出且為可重複模式 ────→ 寫成「降級配方」，降回 Sonnet/Haiku 批次套用
升級到頂仍失敗（共兩輪）═══→ 熔斷，走 judgment-matrix.md §3，問 user
```

**失敗軌跡標準格式**（升級時必附；不附 = Opus 只會重走 Sonnet 的老路）：

```
目標：〈原驗收條件〉
嘗試 1：做了〈操作〉→ 結果〈錯誤訊息關鍵 1-3 行〉
嘗試 2：做了〈操作〉→ 結果〈同上〉
目前假設：〈為什麼失敗的猜測〉
已排除：〈確定不是原因的方向 + 排除依據〉
```

**降級配方格式**（Opus → 便宜模型批次套用時必附）：

```
已驗證解法：在〈範例檔〉做了〈精確改法描述〉，驗證指令〈cmd〉、預期輸出〈...〉。
你的任務：對以下 N 個檔案套用同一模式：〈清單〉。
逐檔套用後跑同一驗證指令；任何一檔結果不同 → 立刻停止並回報，禁止自行變通。
```

## 5. 隔離驗證

**鐵律：實作者不得自我驗證。** 實作 agent 回報「完成」只是「宣稱完成」，狀態仍是未驗證。

原因（寫給弱模型看的機制說明）：實作者的 context 已被自己的假設污染——它驗證的是「我有沒有照**我的理解**做完」，不是「做的東西對不對」。錯誤的理解會同時產生錯誤的實作＋通過的自我驗證，形成閉環。驗證必須來自沒有這份污染的 fresh context。

| 產出型態 | 驗收模式 | 做法 |
|---------|---------|------|
| 文件 / 設定 / 規則檔 | read-back | fresh subagent 只拿驗收條件＋檔案路徑，重新讀檔逐條核對 |
| 程式碼 | 實跑測試 | fresh subagent 實際執行編譯/測試，回報輸出證據；不接受「應該會過」 |
| 設計 / 方案選擇 | 多樣本評審 | 2-3 個獨立 agent 各出方案 → 一個 opus 裁判比較選優 |

驗收 agent 派工鐵則：
- 只給「驗收條件 + 檔案路徑」，**不給實作者的自述**（防錨定：看過「我做完了」的驗證者傾向找通過的理由）。
- 回報必須是 `PASS` / `FAIL` + 逐條證據（`路徑:行號` 或測試輸出末段）。
- FAIL → 回到 §4 狀態機重派實作，**指揮官不自己下場修**。

## 6. 本機工具鏈守則（本機化新增；原全域 CLAUDE.md TOOL-USAGE 段移入）

search-first: 涉及既有程式碼修改前，先 grep/Read 確認使用點，不要憑記憶。

cross-verify: 工具結果與上下文/已知事實矛盾時（如文件引用某路徑但搜尋回空），用第二種工具（ls/find/Bash）交叉驗證再下結論。

edit-tool: 同檔案重複 pattern 修改用 replace_all（避免 context-mismatch error）；single occurrence 才用 old_string/new_string。

upstream-trace: 斷言「沒有 X 檢查/攔截/防線」前，先找當前函式/頁面的所有進入點（誰呼叫它、誰 navigate 過來），往呼叫鏈上游至少追一層；當前檔案內找不到不代表流程中沒有，上游進入點可能早就做過該檢查。grep 進入點 + 讀上游守衛邏輯，是斷言「無防線」的前置必要動作。

verify-vcs-state: 斷言「某 PR/commit/程式碼已在 master/目標分支」前，必須驗 origin/master ancestry（git cat-file / git merge-base --is-ancestor）；working tree 看得到 ≠ 目標分支已有。

graph-first: 在已建 codebase-memory-mcp 索引的專案（`list_projects` 確認）裡，找 symbol 定義/用法優先用 `search_graph`/`search_code` 取代 Grep；追呼叫鏈/評估改動影響面優先用 `trace_path`(direction, risk_labels) 取代手動追呼叫者。`trace_path` 對「方法當 callback 參照傳遞」（React method 綁定後當 prop 傳出、Redux dispatch 等間接呼叫）會漏，callers 回空不能直接當「無人呼叫」的結論，需再用 grep 交叉驗證後才能斷言安全；專案未索引則直接用 Grep/Glob。
