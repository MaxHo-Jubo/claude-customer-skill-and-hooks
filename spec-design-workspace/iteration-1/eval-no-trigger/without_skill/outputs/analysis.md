# 分析：無 Skill 引導下處理「這個 function 太長了，幫我重構」

## 1. 我會怎麼處理這個需求

### 第一反應：資訊不足，需要釐清

這個請求缺少關鍵資訊：**哪個 function？在哪個檔案？** 沒有指向任何具體程式碼。

### 自然處理流程

**Step 1：問清楚目標**

直接回問：

> 哪個 function？給我檔案路徑和函式名稱，或者直接貼程式碼。

不會猜測、不會掃描整個專案找「最長的 function」。使用者說「這個」，代表他心中有明確目標，但沒傳達出來。

**Step 2：拿到程式碼後，讀完整個 function**

用 `Read` 工具讀取該檔案，理解：
- 這個 function 在做什麼（整體職責）
- 有多長（行數）
- 有幾個邏輯區塊
- 有沒有巢狀結構（if/for/try 疊加）
- 參數數量與型別
- 回傳值

**Step 3：分析可拆分的邊界**

找出自然的切割點：
- 獨立的邏輯階段（資料準備、驗證、轉換、呼叫外部、回傳組裝）
- 重複的模式（同樣結構出現多次 → 抽成共用函式）
- 條件分支的處理邏輯（每個 branch 內容夠多 → 各自抽出）
- 副作用與純邏輯的分離

**Step 4：決定重構策略**

依據 CLAUDE.md 的核心原則：
- 超過 3 層縮排 → 重構
- 一個函式只做一件事
- 消除邊界情況 > 增加條件判斷
- 函式 < 50 行

可能的策略：
1. **Extract Method** — 把各邏輯階段拆成獨立函式
2. **Replace Conditional with Polymorphism** — 條件分支太多時考慮
3. **Introduce Parameter Object** — 參數太多時打包
4. **改變資料結構** — 如果 if 太多是因為資料結構不對

**Step 5：執行重構，保持行為不變**

- 逐步拆分，每次只動一個切割
- 確保 function signature 對外不變（向後相容）
- 如果有測試，跑測試確認沒壞
- 如果沒測試，先問要不要補

**Step 6：檢查結果**

- 每個新函式是否職責單一
- 命名是否清楚表達意圖
- 是否符合 STEP 註解規範
- 是否有 JSDoc

## 2. 我的第一個動作/問題

直接問：

> 哪個 function？給我檔案路徑跟函式名稱。

原因：
- 這是 GATE-1 需求確認的「Show me the case」— 不憑空分析
- 沒有具體程式碼，任何分析都是空談
- 不會假裝聰明地掃描專案猜使用者想改什麼

---

## 與有 Skill 引導的差異（預期）

沒有 `/method-refactor` skill 的情況下：
- **沒有結構化的檢查清單**：憑經驗判斷該檢查什麼，可能遺漏
- **沒有固定的輸出格式**：重構結果的呈現方式不一致
- **沒有系統化的品質評分**：「改完了」但沒有量化改善幅度
- **依賴個人判斷**：切割粒度、命名風格、拆分策略都靠臨場決定
- **可能跳過某些面向**：例如忘記檢查 error handling 路徑、忘記看 null safety

整體來說，自然處理會是「能力到位但流程不穩定」— 有經驗的情況下結果不會差，但缺乏可重複的品質保證。
