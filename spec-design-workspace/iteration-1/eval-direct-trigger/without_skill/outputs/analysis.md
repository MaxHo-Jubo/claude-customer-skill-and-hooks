# 無 Skill 引導下處理 `/spec-design 新增使用者匯出 CSV 功能` 的分析

## 情境

使用者輸入 `/spec-design 新增使用者匯出 CSV 功能`，但假設我無法存取 spec-design skill。以下是我在沒有 skill 結構化流程引導下，會如何處理這個需求。

---

## 1. 我會如何切入這個需求？

### 自然反應：直接套用 CLAUDE.md 的需求確認流程（GATE-1）

沒有 skill 引導時，我會回到 CLAUDE.md 定義的基本規則：

1. **自問三個問題**：這是真問題嗎？有更簡方案嗎？會破壞什麼？
2. **一句話重述需求**，請使用者確認
3. **要求具體案例**：誰要匯出？匯出什麼資料？什麼格式？什麼觸發條件？
4. **分析**：資料結構 → 特殊情況 → 複雜度 → 破壞性 → 實用性
5. **輸出**：【核心判斷】值得做/不值得做 →【關鍵洞察】→【方案】

### 實際上我會問的第一個問題

> 「使用者匯出 CSV」— 哪個系統的使用者？匯出什麼資料？是管理後台的管理者匯出使用者列表，還是終端使用者匯出自己的資料？給我一個具體的使用場景。

---

## 2. 第一個動作是什麼？

**問問題，不動手。**

具體來說，我會一次問清楚以下資訊：

- **目標系統**：luna FE？v3 FE？App？
- **使用者角色**：管理員？一般使用者？
- **匯出內容**：哪些欄位？是否涉及敏感資料（個資、聯絡方式）？
- **觸發方式**：按鈕？定時排程？API？
- **資料量級**：幾十筆？幾千筆？幾十萬筆？（決定同步或非同步）
- **既有先例**：專案中有沒有類似的匯出功能可以參考？

---

## 3. 我會遵循什麼結構/流程？

### 沒有 skill 時的自然流程

```
Step 1: 需求確認（GATE-1）
  ├── 問問題釐清範圍
  ├── 重述需求請使用者確認
  └── 判斷值不值得做

Step 2: 判斷是否需要 Plan Mode（GATE-2）
  ├── 匯出 CSV 涉及 3+ 步驟 → 進 plan mode
  └── 寫計畫到 tasks/todo.md

Step 3: 探索 codebase
  ├── grep 專案找既有匯出功能
  ├── 了解資料模型與 API 結構
  └── 找可複用的 pattern

Step 4: 設計方案
  ├── 資料結構先行
  ├── API endpoint 設計
  ├── 前端互動流程
  └── 邊界情況處理

Step 5: 實作（TDD）
  ├── 寫 failing test
  ├── 寫最小實作
  ├── 驗證通過
  └── commit
```

---

## 4. 與 spec-design skill 流程的差異比較

| 面向 | 沒有 skill | 有 spec-design skill |
|------|-----------|---------------------|
| **探索深度** | 依賴個人判斷，可能遺漏面向 | Phase 0 openspec explore 結構化探索問題空間 |
| **收斂方式** | 直覺式收斂，可能跳過方案比較 | Phase 1-4 brainstorming 強制做方案比較與推薦 |
| **產出格式** | tasks/todo.md（簡單 checkbox） | openspec change（proposal + specs + design + tasks + plan） |
| **Review 品質** | 自己 review 自己，盲點多 | 4 個平行 Opus subagent，各有專責面向 |
| **迭代機制** | 無結構化迭代，靠使用者反饋 | 🔴/🟡/🟢 分級，迭代到 🔴=0 才過關 |
| **計畫細度** | 粗略步驟 | Wave 分組 + TDD 步驟 + Model 分級標註 |
| **可接續性** | plan 跟 session 綁定 | plan.md 持久化，可跨 session 用 /plan-and-execute 執行 |
| **風險覆蓋** | 靠經驗想到什麼查什麼 | Agent 3 專責邊界與風險、Agent 4 專責範圍與品味 |

### 關鍵缺失

1. **沒有結構化的多視角 review**：最大差異。一個人想不到 4 個專責 agent 能想到的面向。完整性、可實作性、邊界風險、範圍品味 — 自己做容易顧此失彼。

2. **沒有 openspec 格式約束**：自由格式的設計文件容易遺漏關鍵區段（例如忘記寫 migration 策略、忘記定義 error scenario）。openspec 的 artifact template 強制你填完每個區段。

3. **沒有 plan review 機制**：寫完計畫就直接做，沒有人檢查計畫本身的 TDD 合規性、依賴順序、覆蓋率。

4. **沒有可接續的產出物**：todo.md 是給當下 session 用的，不像 openspec change + plan.md 可以跨 session、跨開發者接續。

---

## 5. 結論

沒有 spec-design skill 時，我仍然能透過 CLAUDE.md 的 GATE-1/GATE-2 規則完成基本的需求確認和計畫制定。但流程會顯著缺乏：

- **系統性**：探索和收斂依賴個人經驗，沒有強制流程
- **review 深度**：自己 review 自己，4 個專責面向退化為 1 個泛用 review
- **產出可持久性**：結果跟 session 綁定，無法結構化接續
- **品質保證**：沒有迭代到 🔴=0 的閘門機制

spec-design skill 的核心價值不是「多了幾個步驟」，而是**強制多視角 review + 結構化迭代**，這是單一 agent 靠自律無法穩定達成的。
