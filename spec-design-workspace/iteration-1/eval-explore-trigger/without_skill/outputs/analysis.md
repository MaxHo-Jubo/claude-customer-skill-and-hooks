# Without Skill: 自然回應分析

## 使用者輸入

> 「我想設計一個新的排班自動分配功能，目前還沒想清楚要怎麼做」

## 1. 我會怎麼處理這個請求

### 觸發的規則判斷

這個請求觸發了 CLAUDE.md 的 **GATE-1 需求確認**：

- 這是一個「收到新任務/需求」的情境
- 需求模糊（「還沒想清楚」）
- 不是簡單明顯修正，不能直接做
- 不是單純問問題或閒聊

同時也觸發了 **GATE-2 Plan Mode**：

- 涉及架構決策
- 需求模糊
- 預期 3+ 步驟

### 自然行為

沒有 skill 引導時，我會依照 CLAUDE.md 的 GATE-1 流程走，但方式是**即興的、線性的**：

1. 自問「真問題？更簡方案？會破壞什麼？」
2. 一句話重述需求，請使用者確認
3. 要求具體案例（Show me the case）
4. 等使用者回答後才能繼續分析

## 2. 我的第一個動作/問題

我會直接回覆使用者，大致如下：

---

確認一下：你想做的是「根據某些規則（人員偏好、法規限制、公平性等），自動把班次分配給可用人員」，對嗎？

在往下走之前，需要你回答幾個問題：

1. **具體場景**：目前排班是怎麼做的？手動？有什麼痛點？
2. **規模**：多少人、多少班次、排班週期（日/週/月）？
3. **限制條件**：有哪些硬性規則？（例如：每人每週不超過 X 小時、不能連續上夜班）
4. **既有系統**：這功能要加在哪個專案上？有既有的排班資料結構嗎？

---

## 3. 我會遵循什麼結構/流程

### 沒有 skill 時的自然流程

```
使用者提需求
    │
    ▼
GATE-1: 需求確認（一句話重述 + 要求案例）
    │
    ▼
等使用者回答
    │
    ▼
分析：資料結構 → 特殊情況 → 複雜度 → 破壞性 → 實用性
    │
    ▼
輸出：【核心判斷】→【關鍵洞察】→【方案】
    │
    ▼
GATE-2: 寫計畫到 tasks/todo.md
    │
    ▼
確認後開工
```

### 問題與限制

1. **沒有結構化的探索框架**：GATE-1 只告訴我「要確認需求」，但沒有提供系統化的 brainstorming 方法。我會憑經驗問問題，但可能遺漏重要面向。

2. **沒有多角度審查機制**：CLAUDE.md 提到可以用 subagent 做平行分析，但沒有具體的 review 流程（例如 4-agent review）。我可能會直接從「問完問題」跳到「寫方案」，中間缺少系統性的挑戰與驗證。

3. **沒有標準化的輸出格式**：最終產出可能是散落的對話，而不是結構化的 spec 文件。要靠使用者自己整理。

4. **探索深度依賴對話輪數**：每個面向都需要一來一回，可能需要很多輪對話才能釐清需求。沒有機制一次性引導使用者思考多個維度。

5. **沒有 openspec 或類似的標準格式**：產出的設計文件沒有固定結構，品質取決於當下的判斷。

### 與有 skill 時的預期差異

| 面向 | 無 skill | 有 skill（spec-design） |
|------|----------|------------------------|
| 探索方式 | 即興提問，線性對話 | 結構化 brainstorming 流程 |
| 涵蓋面向 | 依經驗，可能遺漏 | 預設檢查清單，系統化覆蓋 |
| 審查機制 | 無，或手動 | 4-agent parallel review |
| 輸出格式 | 自由格式 | openspec 標準化格式 |
| 迭代方式 | 口頭修改 | review → 修正 → 再 review |
| 可追溯性 | 散落在對話中 | 獨立 spec 文件，可版控 |

## 總結

沒有 skill 引導時，我會遵循 CLAUDE.md 的 GATE-1/GATE-2 流程，但整個過程是**即興的、依賴經驗的、缺乏系統化結構的**。最大的風險是：

1. 探索不夠全面，遺漏重要面向
2. 缺少多視角的挑戰機制
3. 產出沒有標準格式，難以追溯和迭代
4. 從「需求模糊」到「可執行方案」的路徑不夠清晰
