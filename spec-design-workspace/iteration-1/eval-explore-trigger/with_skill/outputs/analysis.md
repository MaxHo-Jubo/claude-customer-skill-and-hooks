# Spec Design Skill 測試分析：模糊需求觸發行為

## 使用者輸入

> 「我想設計一個新的排班自動分配功能，目前還沒想清楚要怎麼做」

## 1. 應從哪個 Phase 開始，為什麼

**應從 Phase 0: Explore 開始。**

原因：

- 使用者明確說「還沒想清楚」，代表需求處於模糊狀態，問題空間尚未定義。
- SKILL.md Phase 0 的跳過條件是「使用者帶著明確需求直接來（如 `/spec-design 新增使用者匯出功能`）」。本案例不符合跳過條件——使用者沒有明確的功能定義，只有一個模糊方向。
- Phase 0 的定位是「thinking partner，不是工作流」，正好適合這種探索階段。
- Phase 1-4（brainstorming）需要有一定的問題理解基礎才能結構化收斂，直接跳進去會導致在錯誤的問題空間上收斂。

## 2. 在 Phase 0 會做什麼

根據 SKILL.md，Phase 0 的行為定義：

### 會做的事

1. **問問題、挑戰假設、重新定義問題**：
   - 釐清「排班」指的是什麼排班（護理班？工廠輪班？門市排班？）
   - 「自動分配」的範圍（完全自動？建議後人工確認？）
   - 目前是怎麼排班的？痛點在哪？
   - 誰在用？使用頻率？

2. **讀 codebase 了解現有架構**（grounded，不空談）：
   - 檢查專案中是否已有排班相關的資料結構、模組、API
   - 了解現有的使用者/員工/班別 model
   - 確認技術棧限制

3. **畫 ASCII 圖視覺化**：
   - 現有排班流程的狀態機
   - 自動分配的可能資料流
   - 不同方案的比較表

4. **比較方案、發散思考**：
   - 貪心演算法 vs 約束求解 vs 規則引擎
   - 即時分配 vs 批次分配
   - 硬約束（法規、資格）vs 軟約束（偏好、公平性）

5. **若專案已有 openspec change**，讀取既有 artifact 作為探索上下文。

### 不會做的事

- 不寫程式碼、不實作功能
- 不走固定流程、不強制產出
- 不急著收斂

### 結束時機

- 使用者說「夠了」「可以開始了」「進入 brainstorming」
- 或問題空間已充分理解，自然過渡到 Phase 1

## 3. 第一個行動 / 對使用者的第一個問題

根據 CLAUDE.md 的 GATE-1 需求確認流程，結合 Phase 0 的 explore 姿態，第一個行動應該是：

### 一句話重述需求 + 問具體案例

大致會這樣回應：

---

「你想做一個排班自動分配功能，但問題空間還不清楚。在開始探索之前，先確認幾個基本問題：

1. **哪個專案的排班？** 居服 App？日照 App？還是 luna 後台？
2. **Show me the case**：能不能描述一個具體的排班場景？例如：「護理長每週一早上要排下週 30 個居服員的班，目前用 Excel 手動排，痛點是 ___」
3. **「自動」的程度**：是系統全自動產出完整排班表？還是系統建議、人工微調？

先搞清楚是什麼問題，再來想怎麼解。」

---

### 為什麼是這個問題而不是別的

- GATE-1 要求「Show me the case：要求具體案例」，不憑空分析。
- Phase 0 的核心是「問問題、挑戰假設、重新定義問題」。
- 從 PROJECT-MAP 看，使用者有多個專案（居服/日照/luna），不確認目標專案會導致後續探索方向錯誤。
- 「自動」是一個高度模糊的詞，需要先界定範圍才能有效探索。

## 4. 整體流程預期路徑

```
Phase 0: Explore（預計 2-5 輪對話）
  - 釐清問題空間、痛點、現有流程
  - 讀 codebase 了解現有排班相關程式碼
  - 視覺化現有流程與可能方案
    ↓
Phase 1-4: Brainstorming（調用 superpowers:brainstorming skill）
  - 結構化收斂：需求釐清 + 方案比較 + 設計確認
    ↓
Phase 5: 撰寫 openspec
    ↓
Phase 6-7: 4-agent spec review + 迭代修正
    ↓
Phase 8-9: Plan mode + 4-agent plan review
    ↓
Phase 10: 使用者確認 + 轉入 plan-and-execute
```

## 5. Skill 在此場景的關鍵價值

- **防止過早收斂**：沒有 skill 的話，容易直接開始討論技術方案（演算法選擇等），跳過問題定義。Phase 0 強制先探索問題空間。
- **結構化推進**：從 explore → brainstorming → spec → review → plan，每個階段有明確的進入/退出條件，不會在模糊狀態下就開始寫 code。
- **GATE-1 整合**：Skill 的 Phase 0 與 CLAUDE.md 的 GATE-1 需求確認流程天然互補——都要求先確認「這是真問題嗎」「Show me the case」。
