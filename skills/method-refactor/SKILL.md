# Method Refactor

對指定方法進行結構化優化重構，逐項檢查並改善程式碼品質。

## 使用方式

- `/method-refactor <方法名稱或檔案路徑:行號>` — 完整重構流程
- 使用者可指定只做部分檢查項目

## 檢查清單

依序執行以下 7 項檢查，每項標記 ✅ 已優化 / ⏭️ 不適用 / ❌ 有問題待討論：

### 1. 常數掃描

- 找出方法內的硬編碼字串與數字（magic numbers / magic strings）
- 搜尋專案現有的 Constants 檔案，確認是否有對應常數可替換
- 有對應常數 → 替換
- 無對應但多處使用 → 建議新增常數

### 2. 型別常數化

- 找出 library API 的固定參數（如日期格式、時間單位等）
- 用 `as const` 宣告模組級常數，確保型別安全
- 同一常數在檔案中出現多處時，全檔統一替換

### 3. 邏輯扁平化

- 巢狀超過 2 層的 if/else → 用 early return 攤平
- `if (!(a === b))` → `if (a !== b)`
- `if (cond) { return true; } return false;` → `return cond;`
- 反轉條件讓 happy path 在最外層

### 4. Promise → async/await

- 識別 `return new Promise((resolve, reject) => {...})` 模式
- **前提檢查**：grep 所有外部呼叫方，確認改成 async 後回傳型別不變
- 若外部呼叫方需要改動 → 報告給使用者，不自動變更
- 轉換後消除 `.then()/.catch()` 回調地獄

### 5. 重複碼提取

- 相同邏輯出現 2 次以上 → 提取為 helper function 或 class method
- 命名遵循專案既有的命名慣例
- helper 放在使用處的正上方，附上用途註解

### 6. 冗餘操作移除

- 識別對已處理過的值重複操作（如對已轉換過的值再做一次相同轉換）
- 移除冗餘操作
- **必須加 NOTE 註解**說明為何移除，格式：
  ```
  // NOTE: 原本此處對 xxx 執行 yyy，但 xxx 已在 STEP XX 做過相同處理，故直接使用。
  ```

### 7. 測試驗證

- 每個方法改完後立即跑對應的測試
- 確認全部通過才進入下一個方法
- 測試失敗 → 修復或回滾，不帶著紅燈繼續
- **Linter 檢查**：對修改的檔案執行 `npx eslint <file>`（若專案有 ESLint），確認無新增 lint 錯誤
- **型別檢查**：若目標檔案是 `.ts` / `.tsx`，額外執行 `npx tsc --noEmit <本次重構變更的檔案>` 確認無型別錯誤；`.js` / `.jsx` 檔案跳過此步驟（tsc 對無型別標註的 JS 檢查噪音大、價值低）

## 輸出格式

每個方法完成後輸出摘要：

```
### methodName 優化摘要

| # | 檢查項 | 結果 | 說明 |
|---|--------|------|------|
| 1 | 常數掃描 | ✅ | 替換了 N 個硬編碼值 |
| 2 | 型別常數化 | ⏭️ | 無需新增 |
| 3 | 邏輯扁平化 | ✅ | 反轉巢狀 if，early return |
| 4 | Promise→async/await | ✅ | 外部呼叫方不受影響 |
| 5 | 重複碼提取 | ✅ | 提取 helperName() |
| 6 | 冗餘移除 | ✅ | 移除重複的 xxx 操作 |
| 7 | 測試驗證 | ✅ | N/N passed |
```

## 注意事項

- **不改邏輯**：重構不改變行為，只改結構
- **不改 typo**：方法名稱的拼字錯誤不主動修正，除非使用者明確要求
- **刪檔前問**：任何檔案刪除都要先確認使用者意圖
- **一次一個方法**：改完、測完、確認 green，再進下一個
- **commit 粒度**：每個方法（或相關方法群組）一個 commit
