# Linus Style 需求分析

對需求進行結構化審查，輸出明確的做/不做判斷。

## 使用方式

- `/linus-requirements-analysis` — 對當前 context 中的需求執行分析
- `/linus-requirements-analysis {需求描述}` — 對指定需求執行分析

## 輸入來源

依優先序：
1. 使用者直接提供的需求描述
2. 當前 `.claude/{ISSUE_ID}-Jira.md` 中的需求內容（從 `/jira fetch` 取得）
3. 使用者口頭描述 → 要求補充具體案例

## 分析框架

依序執行以下 6 步，每步產出一段結論：

### STEP 1: 這是真問題嗎？

區分三種來源，判斷標準不同：
- **法規合規** → 必須做，不討論
- **真實痛點** → 有具體案例、有頻率、有影響範圍 → 值得做
- **想像中的問題** → 「萬一…」「以後可能…」→ 不做

### STEP 2: Show me the case

要求具體案例，不憑空分析：
- 截圖或畫面描述
- 業務邏輯（觸發條件、前置狀態）
- 必填規則、驗證邏輯
- 實際發生的頻率與影響人數

如果提供的需求缺少以上資訊，**停下來要求補充**，不猜測。

### STEP 3: 資料結構先行

先定義 data model，再討論 UI 和流程：
- 這個需求影響哪些資料結構？
- 欄位掛在哪個層級？（plan-level vs item-level、entity vs relation）
- 現有 schema 能不能直接支撐，還是要改？

### STEP 4: 有更簡單的方案嗎？

- 能不能用現有功能組合解決，而不是新開發？
- 能不能改設定、改流程、改文件，而不是改程式碼？
- 最小改動量是什麼？

### STEP 5: 會破壞什麼？

- 現有資料相容性（舊資料跑新邏輯會不會炸）
- 其他客戶 / 模組的影響
- 向後相容：破壞既有功能 = bug

### STEP 6: 不要過度泛化

- 先解決眼前的具體案例
- 不做「順便支援」、「未來可能需要」的擴充
- 一個需求解決一個問題

## 輸出格式

```
【核心判斷】值得做 / 不值得做 / 需要更多資訊

【關鍵洞察】
- （1-3 條最重要的發現）

【資料結構影響】
- （受影響的 model / schema / 欄位）

【方案】
- （具體做法，含最小改動路徑）

【風險】
- （破壞性、相容性、影響範圍）

【不做的事】
- （明確列出本次不處理的擴充或泛化需求）
```

## Jira 回寫（選用）

分析完成後，如果當前有對應的 Jira issue，詢問使用者是否將分析結論回寫為 Jira comment：

```
需求分析完成。是否將結論回寫到 {ISSUE_ID} 的 Jira comment？(y/n)
```

若同意，使用 Atlassian MCP 工具 `addJiraComment` 將輸出格式的內容寫入 issue comment。
